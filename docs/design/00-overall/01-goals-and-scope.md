# 01 · 目标、范围与术语（总体设计）

> 第一册《总体设计》· 第 1 篇

> 上游依据：`信道模拟器3协议V3.0.docx`、`MPDB接口的使用.md`、兄弟项目 `ChannelEgine`

---

## 1. 概述

本项目把现有的 `channel_simulator` 离线工具链，演进为**可商业运行的信道仿真控制平台（Channel Emulation Platform, CEP）**。

平台的一句话定位：

> **把「信道模型」与「信道仿真硬件」解耦——上游接入多种信道模型来源（统计模型 / 确定性射线），中间收敛为一份与设备无关的统一信道表示，下游渲染并下发到多种仿真器后端——并以分层 API 同时服务自有软件与第三方。**

其中的旗舰能力：**将 RT 射线法（经 MPDB）的计算结果装入仿真器「播放」，首先以 TDL 形态呈现，并具备 MIMO 空间相关矩阵（Kronecker）。**

本篇只界定**做什么 / 不做什么 / 名词含义**，为后续各篇（架构、API、数据流、相关性等）提供统一的范围基线。

## 2. 产品目标（Goals）

| 编号 | 目标 | 衡量方式（验收信号） |
| :--- | :--- | :--- |
| G1 | **统计信道编辑**：支持标准 3GPP TR 38.901 系列信道（UMa/UMi/RMa/InH、CDL/TDL、LSP、簇、XPR、K 因子）的生成与编辑。 | 复用 `ChannelEgine` 引擎产出信道，可经平台下发到设备。 |
| G2 | **RT→TDL 播放（旗舰）**：从 MPDB 读取确定性射线结果，量化合并为 ≤24 径 TDL，并具备 MIMO 空间相关矩阵。 | 一条 MPDB 链路端到端跑通：MPDB → TDL(带相关) → 设备下发 → 遥测确认。 |
| G3 | **硬件控制**：完整覆盖协议 V3.0 的下行控制与上行遥测；事务化、可校验地驱动 RF-SoC 设备。 | 35 个参数 ID 全部可编码；4 类下行帧可解析；下发有复制帧/遥测闭环确认。 |
| G4 | **多后端下发**：同一份信道模型可下发到 ①RF-SoC 二进制协议设备（以太网）②`.asc` 时变 CIR 回放设备。 | 同一 canonical model 渲染出两种后端产物且各自可被目标设备接收。 |
| G5 | **分层 API**：REST（主）+ Python SDK + SCPI 兼容层，既服务自有 GUI，也服务第三方/自动化。 | 三种前端调用同一服务层完成「加载场景→下发→查遥测」闭环。 |
| G6 | **商业级非功能**：可靠（事务/回滚/重连）、可观测（遥测指标/告警）、可审计、可校准、配置即数据（可序列化/版本化/重放）。 | 见《11-nfr-cross-cutting》各项验收。 |
| G7 | **多层级输入**：允许用户在信道模型嵌套的任一层输入（RT⊃GCM⊃CDL⊃TDL），平台沿退化链降到硬件可实现层级，由 RF-SoC 实现。 | 见《03b-model-hierarchy》；四层入口均可导向下发/实现。 |

## 3. 范围（In Scope）

本期（P0–P4，见《13-roadmap》）平台**自建并负责**：

- **L1 编解码**：协议 V3.0 二进制帧的全 ID 编码 + 4 类下行帧解码；`.asc` CIR 读写；MPDB 读取；单位换算。
- **L2 设备后端抽象**：`DeviceBackend` 接口 + `RFSoCBackend`（帧/TCP/回显校验/重连）+ `AscCirBackend`（时变 CIR→.asc）。
- **L3 服务/编排**：信道模型引擎集成（ChannelEgine）、RT/MPDB 导入管线、相关性合成、场景/会话状态机、遥测采集、校准、**统一内部信道表示（canonical model）**。
- **L4 API 网关**：REST/OpenAPI + Python SDK + SCPI-over-TCP。
- **L5 客户端**：Python SDK；**自有 GUI 全新设计**（不复用 ChannelEgine 的 `gui.py`）。GUI 仅通过平台服务层/ API 调用 ChannelEgine 的引擎能力，不依赖其界面。

---

## 4. 不做 / 外部依赖（Out of Scope）

| 项 | 说明 | 归属 |
| :--- | :--- | :--- |
| RT 射线追踪本体 | 平台**不做**光线追踪计算，只消费其产物（MPDB）。 | HyperRT SDK（外部） |
| 38.901 模型算法本体 | 平台**不重造**统计信道算法，集成 `ChannelEgine`。 | ChannelEgine（兄弟项目，独立 git） |
| RF 前端 / FPGA 固件 | 平台经协议透传（ID30/35）对接 RF 前端，**不开发**其固件；实时衰落在 RF-SoC/FPGA 内生成。 | 硬件团队 |
| 真实 RF 定标测量 | 平台做数值域校准（dB/量化/溢出保护），**不含**接矢网/功率计的物理定标流程。 | 硬件/测试团队 |

---

## 5. 非目标（Non-Goals）

- **不承诺协议未暴露的能力**。协议 V3.0 无空间相关字段、瑞利逐信道逐径独立，因此本期**不承诺硬件级瞬时 MIMO 空间相关**（详见《06-correlation-mimo》与《12-risks-open-items》）。当期以「统计等效（B 档）」交付，「时变 CIR 注入（A 档）」作为预留接口，待硬件确认后再定档。
- **不做通用网络/系统级仿真器**；平台聚焦物理层信道回放与设备控制。
- **不在本期锁定 gRPC 流式接口**（列为可选，P2 之后按需）。

---

## 6. 关键约束（Constraints，源自协议/数据事实）

这些是硬约束，后续所有设计不得违背（出处见《03-architecture》与各专篇）：

- **信道栅格**：8×8 = 64 信道（输入端 0–7 × 输出端 0–7）。
- **每信道多径**：24 条；24 条均支持相偏。
- **主时延**：码值 0–1050，单位 `1000/120 ns`。
- **控制帧 payload 上限**：4000 字节（不含头/长度/尾）——大配置须分帧。
- **瑞利频域滤波系数**：1024 字节 = 256 复数，长度字段写 0 的特例；**仅定义多普勒谱形，不注入指定相干实现**。
- **MPDB 坐标**：天顶角（Zenith），下发前需换算仰角 `Elevation = 90° − Zenith`；时延秒→纳秒。
- **传输**：以太网 / TCP（非串口）。

## 7. 术语表（Glossary）

| 术语 | 含义 |
| :--- | :--- |
| **CEP** | Channel Emulation Platform，本平台。 |
| **canonical model** | 统一内部信道表示；与设备无关的信道描述（每信道对的 TDL 抽头集：时延/复增益/多普勒/相关信息，及可选时变 CIR）。全系统枢纽契约。 |
| **TDL** | Tapped Delay Line，抽头延迟线信道模型。 |
| **CIR** | Channel Impulse Response，信道冲激响应；此处特指逐时刻时变复抽头序列（`.asc` 的内容）。 |
| **MPDB** | Multipath Database，HyperRT 的射线多径数据库（PyTorch 张量），本平台 RT 输入来源。 |
| **MPQL** | MPDB 的 SQL 风格查询语言。 |
| **ChannelEgine** | 兄弟项目：3GPP TR 38.901 信道模型引擎，本平台上游统计信道来源。**仅集成其引擎/服务能力，不复用其 GUI。** |
| **DeviceBackend** | 设备后端抽象接口；实现有 `RFSoCBackend`、`AscCirBackend`。 |
| **RFSoCBackend** | 把 canonical model 组装为协议 V3.0 二进制帧并经 TCP 下发的后端。 |
| **AscCirBackend** | 把 canonical model 渲染为 `.asc` 时变 CIR 文件的后端。 |
| **CorrelationSynthesizer** | 相关性合成策略接口；B 档（统计等效）为默认实现，A 档（时变 CIR）为预留实现。 |
| **A 档 / B 档** | MIMO 相关性两种实现哲学：A=真实瞬时相关（需硬件 CIR 注入）；B=统计等效（几何/确定性相关精确复现 + 衰落边缘统计匹配）。 |
| **子帧 / SubFrame** | 协议中一个参数单元：`ID(1)+长度(1)+IO(1)+补充信息(1)+值`。 |
| **IO 字节** | 输入输出端编码：高 4bit=输入端，低 4bit=输出端。 |
| **info / 补充信息** | 子帧第 4 字节，通常为多径编号（0 对应「多径1」）。 |
| **ChannelAddress** | 统一寻址元组 `(device_id, input, output, path)`，支撑多设备扩展。 |

## 8. 与现有代码的关系（起点盘点）

- 现 `ChannelSimulationCode/channel_simulator/` 已实现：协议 V3.0 部分编码（13/35 ID）、RT→TDL 量化合并、TDL→帧组装、CSV/BIN/HEX 输出。**它是 L1 的一部分与 L2/L3 的雏形。**
- 已知缺口（后续篇章细化）：串口/网络**下发能力缺失**（重构退化，需以 TCP 找回）；下行帧解码缺失；瑞利/扫频/信号源/拓扑等 ID 未接线；MPDB 读取、多后端、相关性合成、API 层均未建。
- 遗留单文件 `ChannelSimulationCode/Code` 仅作历史对照，不在其上开发。

---

## 9. 开放问题（本篇相关，汇总见《12-risks-open-items》）

1. RF-SoC 是否支持时变 CIR/抽头流式注入（决定 A 档能否承诺）。
2. `.asc` 后端的目标设备是 RF-SoC 的 CIR 回放模式，还是第三方（PropSim/Spirent）。
3. MPDB 运行是否随产品部署 HyperRT SDK，还是走 MPQL 导出 CSV 解耦。
