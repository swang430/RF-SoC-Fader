## 1. 概述

本篇从**外部视角**界定平台边界：谁在用、连接哪些外部系统、数据从哪来到哪去、支撑哪些顶层用例。内部分层留给《03-architecture》。

一句话上下文：

> 平台居于「**信道模型来源**」与「**信道仿真硬件**」之间，向上以 API 服务人和软件，向下以多后端驱动设备，把 RT/统计两类模型统一「播放」到仿真器上。

## 2. 使用者画像（Personas）

| 角色 | 目标 | 主要入口 | 关注点 |
| :--- | :--- | :--- | :--- |
| **信道工程师** | 编辑/加载信道场景，下发设备，看效果 | 自有 GUI（全新设计） | 直观、可视化 PDP/相关、所见即所得 |
| **测试自动化工程师** | 把信道配置纳入自动化测试台，批量/回归 | Python SDK、SCPI-over-TCP | 可脚本化、可复现、与测试框架对接 |
| **第三方集成方** | 在自有软件里调用信道仿真能力 | REST/OpenAPI | 稳定契约、语言无关、鉴权清晰 |
| **RT/算法研究者** | 把射线仿真（MPDB）结果装进硬件播放 | REST/SDK（导入 MPDB） | 保真度、相关性、单位/坐标正确 |
| **运维/现场** | 部署、监控设备健康、排障 | 遥测/告警、审计日志 | 可观测、可靠、可回滚 |

## 3. 外部系统与接口（External Interfaces）

| 外部系统 | 方向 | 接口/协议 | 内容 | 归属 |
| :--- | :--- | :--- | :--- | :--- |
| **HyperRT / MPDB** | 入 | Python `MPDB.load()`（PyTorch 张量）或 MPQL 导出 CSV | 射线多径：DELAY/H/AoA/ZoA/AoD/ZoD/天线坐标 | 外部 SDK |
| **ChannelEgine 引擎** | 入 | 服务/函数调用（`ChannelSimulator` 类等） | 38.901 信道：LSP/簇/角度/功率/时延/XPR，及时变 CIR | 兄弟项目（集成，**不含其 GUI**） |
| **RF-SoC 仿真器** | 出/入 | 协议 V3.0 二进制帧 over **TCP** | 下行控制帧；上行遥测/复制/错误帧 | 硬件团队 |
| **.asc CIR 回放设备** | 出 | `.asc` 文件（PropSim/Spirent 类） | 逐时刻时变复抽头 CIR | 第三方/待定（见《12》） |
| **RF 前端** | 出/入 | 经协议透传子帧（ID30/35）中继串口 | 控频/衰减/偏置字符串 | 硬件团队 |
| **第三方软件 / 测试台** | 入 | REST / SCPI-over-TCP / SDK | 场景/信道配置、下发指令、遥测查询 | 用户/集成方 |

## 6. 数据流向总览（谁产、谁传、谁存）

- **产**：模型数据由 ChannelEgine / MPDB 产生；控制帧由 RFSoCBackend 产生；`.asc` 由 AscCirBackend 产生。
- **传**：TCP（设备控制/遥测）；文件（`.asc`）；进程内/服务调用（引擎集成）。
- **存**：场景/信道/相关配置以 JSON/YAML 持久化（配置即数据，可版本化）；下发审计日志；遥测时序（指标）。
- **单位/坐标归一**发生在**导入边界**（MPDB s→ns、Zenith→Elevation）与**编码边界**（物理量→协议码值），集中管理不散落。

---

## 7. 信任边界与安全上下文（详见《11》）

- **对外 API（L4）** 是主要信任边界：需鉴权、限流、审计。
- **设备链路（L2/L1）** 直接影响物理射频输出：所有下发进审计日志；错误帧/溢出遥测须可观测。
- **外部数据（MPDB/场景文件）** 视为不可信输入：导入时做 schema 校验与范围检查。
