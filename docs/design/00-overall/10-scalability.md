# 10 · 可扩展架构：单机 → 多设备（总体设计）

> 第一册《总体设计》· 第 10 篇 · 状态：草稿 v0.1
> 前置：《03-architecture》

---

## 1. 概述

本期交付**单机**（一台 RF-SoC，8×8=64 信道），但架构**预留多设备/多机箱扩展**，横向扩展只增寻址与注册层，不改 L1/L3 核心。

---

## 2. 统一寻址

```
ChannelAddress = (device_id, input, output, path)
                    │          └── 8×8 栅格内 ──┘  └ 24 径
                    └── 多设备扩展点（单机时默认单一 device_id）
```
- 单机：`device_id` 恒定/隐式；API `/channels` 等在默认设备上操作。
- 多机：`/devices/{id}/channels`，会话可跨设备编排。

---

## 3. 扩展点

| 层 | 扩展机制 | 单机表现 |
| :--- | :--- | :--- |
| L2 | `DeviceBackend` 持 `transport`（TCP endpoint）与 `device_id` | 单一 endpoint |
| L3 | `DeviceRegistry` 管理多设备生命周期/健康 | 注册单设备 |
| L3 | 会话可绑定多设备（大规模 MIMO 拼装 / 多机箱级联） | 单设备会话 |
| L4 | 资源路径 `/devices/{id}/...` | id 可省略 → 默认设备 |

---

## 4. 两个正交扩展维度

- **后端类型 × 设备实例**：`(rfsoc|asc) × device_id`。
- **信道规模**：单设备 64 信道 → 多设备 N×64；拓扑（8×8/4×8）+ 路径扩展（ID16）在设备内扩展多径。
- 二者正交，组合不产生逻辑分叉（canonical model 与后端接口不变）。

---

## 5. 伸缩性

- **ChannelEgine 微服务**可独立扩容（计算密集）。
- **CEP 主服务**为控制面，单实例可管多设备；多实例需协调设备归属（后续，避免双写同一设备）。
- 遥测采集随设备数线性增长，采用异步轮询/订阅。

## 6. 开放问题
1. 多 CEP 实例下的设备归属与互斥（设备级锁/租约）。
2. 跨设备会话的事务一致性（部分设备失败的整体回滚语义）。

## 7. 本篇验收
- `ChannelAddress` 贯穿 L2/L3/L4，单机为其退化特例。
- 多设备扩展不触碰 L1 与 canonical model。
