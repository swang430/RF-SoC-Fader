# 12 · 设计风险与待确认清单（总体设计）

## 1. 头号风险 ★ — A 档（瞬时 MIMO 相关）可行性

- **问题**：RF-SoC 是否支持「时变 CIR / 相干抽头流式注入 / 相干种子」？协议 V3.0 未暴露此能力。
- **设计方向（已定）**：A 档按**时变相干矩阵输入**设计，目标 = **RF-SoC 自身 CIR 回放模式**（见 #3），`.asc` 作为**预留接口**。canonical model 的 `time.mode=time_varying` + `cir` 承载。
- **仍待确认**：RF-SoC CIR 回放模式的**注入帧/接口是否存在**（协议 V3.0 未暴露）——这是 A 档能否落地的唯一硬门槛。
- **影响**：决定旗舰对外能承诺到哪一档。硬件确认前守卫拒绝 `correlation.mode=A` 实际下发。
- **需谁确认**：硬件/固件团队（**action**）。

## 2. ★ ID6 瑞利谱系数精确语义 + 瑞利功率下降
- **机理已明**（启用 ID5 功率下降）：定点数据通路为避免瑞利高峰均比信号削顶，须预留幅度余量 → 平均输出功率下降；下降量取决于 256 个频域系数的整体归一化与时域过程方差。
- **仍待确认**：256 复数 ↔ 多普勒谱形/最大多普勒的确切映射；各谱形/配置对应的**确切 dB 下降量标定表**。
- 影响：多普勒谱配置正确性、B 档衰落边缘统计、功率补偿校准（《11》§4，校准模块 M8 按标定表补偿）。
- 需硬件标定表/说明（**action**）。

## 3. .asc 后端目标设备与格式版本 —— **已定**
- **目标 = RF-SoC 自身 CIR 回放模式**（非 Keysight PropSim / 非 Spirent）。
- `.asc` 采用 ChannelEgine 现有格式（header + tap data，见《08》§4）作为回放接口；A 档经此在**同一 RF-SoC** 上落地。
- 仍需锁定 RF-SoC CIR 模式对 `.asc`/CIR 帧的确切接收字段（并入 #1 硬件确认）。

## 4. MPDB 运行依赖与接入方式 —— **已定**
- **HyperRT 直接导入，无中间件**（不走 MPQL 导出 CSV）。
- 后果：导入侧依赖 **HyperRT SDK + PyTorch**（`mpdb_reader` 直接 `MPDB.load()`）。
- 影响：《05》§2 只保留 SDK 直读路径；《09》主进程依赖需含 HyperRT/PyTorch（或将导入置于独立进程/服务隔离，待《03》§6 细化）。

## 5. 帧长上限 4000B 的分帧策略
- 64×24×多参数 + 每径 1024B 瑞利系数远超单帧上限。
- 影响：RFSoCBackend 分帧与事务边界（《08》《11》）；需定分帧规则与设备对多帧的接收语义。

## 6. 阵列几何 ↔ 8×8 栅格映射 —— **重要设计，归 M4（RT-MPDB 导入）完善**
- 天线阵元 ↔ 输入/输出端口（0–7）的映射规则、拓扑（8×8/4×8）与路径扩展的消费方式。
- **定位**：这是决定 MIMO 保真的重要设计，本期**留给 RT-MPDB 导入模块（第二册 M4）细化**（`map_links_to_channel_pairs`）。《05》§3、《06》§6 共用此映射定义。

## 7. RT 多普勒来源
- 单 MPDB 快照无多普勒；需几何速度注入或多快照序列（《05》§5）。A 档时间维依赖此。

## 8. 架构内部待定（不阻塞旗舰，团队内决）
- ChannelEngineClient 传输/序列化（gRPC/REST/MQ）+ 张量编码（《07》）。
- 事务/分帧边界归 L2 还是 L3（《11》§7）。
- canonical model 交叉极化（VV/VH/HV/HH）表示与设备承载（《05》《06》）。
- 多 CEP 实例下设备归属互斥（《10》）。
- 遥测订阅 SSE vs gRPC 流（《04》）。
