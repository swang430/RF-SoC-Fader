## 0. 结论速览

| # | 问题 | 一句话结论 |
| :-- | :-- | :-- |
| Q0 | 相位连续性 | A 档：连续性**在数据里**（逐时刻相干 `gain_series`），责任在生成方；B 档：**不承诺跨下发连续**（apply **视同**信道重配置——契约表述，不依赖硬件实际行为），承诺的是**同种子可复现** |
| Q1 | 生灭问题 | canonical 层无生灭字段：时变下 tap 槽位固定，生灭=**增益包络起零/归零**（生成方 ramp）；参数流式面的槽位稳定纪律为保留接口设计 |
| Q2 | 轨迹提取 | 两档出口：A 档=统一时延栅格+逐时刻 `gain_series`→`.asc`（**不做 24 截断**——径数上限属设备面、由能力门按后端裁决；时延冻结带边界，入 FidelityReport）；B 档=快照点列逐次 re-apply（准静态，标注保真损失）；本期不实现（《T1-05》§5-3 既定推迟） |
| Q3 | M4 合并策略 | M4 没有「簇合并」（聚簇归 M5）；M4 是**射线→时延 bin 量化合并**：导向后才合并、bin 内恒相干叠加、power_mode 只影响排序功率、归一化基准外置 |
| Q4 | API 位置 | 不在设备内。设备内只有私有二进制帧协议（不对外）；平台是**外置控制平面**（L4，外部控制机），SCPI 兼容层在平台层复刻「仪器内嵌 API」体验 |
| Q5 | SDK 与 API | SDK（`cep_sdk`）= REST `/v1` 的**薄客户端封装**：唯一上游是 OpenAPI 合同，不直连 L3，零业务逻辑，能力与 REST 一一对应 |

## 1. Q0 · 不同 TDL 之间的相位连续性

**问**：信道随时间演化（或先后下发多套 TDL）时，相邻 TDL 之间的相位如何衔接？

**已有约定**：
- 时变承载：`time.mode=time_varying` 时抽头增益为逐时刻复序列 `gain_series: Complex[n_snapshots]`（隶属 `time.snapshots`），`realization=CIR`（《T1-03c》§5.2/§7）。tap 无独立 phase 字段——相位即 `arg(gain)`（《T1-03c》§5.2）。
- A 档定义即「**逐时刻相干** CIR」（《T1-08》§4）；快照时刻 = `s/update_rate_hz`（t₀=0，《T2-03》§2）。
- B 档下发是事务：reset → 清 24 径 → 逐径写 → 复制帧比对 → 遥测确认（《T1-08》§3；另有 `reset_first=False` 的免复位形态，《T2-02》§3）；若写了多普勒，事务尾部下发**频偏相位归零**（ID33「多径同步起相」，《T2-02》§3、《T1-A1》§4）；瞬时衰落由硬件按参数生成（瑞利使能 ID5 + 频域谱系数 ID6 定多普勒谱形 + 多普勒 ID4，《T1-A1》§4），平台只写参数。
- 初相确定性：cluster_phase 三级优先（①`Cluster.phase_rad` ②v1.0 载体 ③`cluster_phase_seed` 兜底），同 seed+config → 逐比特相同（《T2-05》§3、《T2-03》§2 确定性契约）。

**★裁定（相位连续性责任分配）**：
1. **A 档（CIR 回放面）：连续性在数据里，责任在生成方。** 快照间相位演化（多普勒即相位差分）由生成侧（M3 引擎 / 未来 M4 多快照导入）在产出 `gain_series` 时保证相干；canonical model 只**承运**该序列（不插值、不重相位化）；后端（AscCirBackend）原样落盘、回放设备按 `update_rate_hz` 顺序播放。配套约束：**采样充分性** `update_rate_hz ≥ 2·max|f_d|`（奈奎斯特下限，防快照间相位卷绕假频）——生成方校验，违反拒绝或显式警告。
2. **B 档（参数化 TDL 面）：平台契约不承诺跨下发的相位连续。** 每次 apply/re-apply **视同**信道重配置——不承诺任何相位/衰落状态跨下发保持（等价于换一次信道随机实现，即 3GPP 仿真语境的换 drop）。契约依据：协议无「保持衰落状态热切换」语义；RESET 是参数复位且明文**保留瑞利系数**（状态行为存在未定面，《T2-02》§8 开放问题「RESET 残留」）；ID33 仅是条件性下发的多普勒相位同步。硬件重写时是否实际复位衰落过程属待实测项（D2）——平台契约刻意不依赖其任一答案。
3. **B 档承诺的是可复现，不是连续。** 单次下发内：各径导向相位差（`arg(gain)` 的确定性分量）精确复现（《T2-05》B 档定义）；跨下发：同 seed+同 config 的初相逐比特重复（确定性契约）。需要「跨时刻连续演化」的场景属 A 档诉求——当前 RF-SoC `supports_cir=False`（《T2-02》§2），现实出口 `.asc`；按通用 fader 原则，接口保留、按 capabilities 启用。

## 2. Q1 · TDL 模型之间有没有生灭问题？

**问**：时间演化中路径出现/消失（birth/death），TDL 表示与设备下发怎么处理？

**已有约定**：
- schema 无生灭概念：time_varying 下 `gain_series` 定长 `Complex[n_snapshots]`，taps 集合跨快照**隐含固定**；引擎侧 `delays_s` 明确「快照间不变——时变时延本期不支持」（《T2-03》§2）。
- session 运行态只有 tweak / re-apply，无时间轴播放器（《T2-06》§3/§4bis）；时变逐快照流式下发 RF-SoC 是开放问题（《T1-08》§7）。

**★裁定（生灭的表示与处置）**：
1. **canonical 层不设生灭字段**：时变模型 tap 槽位固定（与 `.asc` 每 CIR 固定 T 抽头一致），生灭以**增益包络**表达——birth = `gain_series` 从 0 起、death = 归零后保持 0。斜坡淡入/淡出（ramp）由生成方负责（防幅度阶跃造成频谱溅射）；schema 字段形态不变。
2. **静态 B 档无生灭问题**：单快照无时间维；两个独立 TDL 模型间的切换是**重配置**（Q0 裁定 2），不属生灭。
3. **参数流式面的槽位稳定纪律（保留接口设计，本期不实现）**：若未来硬件支持逐快照参数流（B 档时变），相邻快照独立做 top-24 选径会导致槽位漂移（同一物理径落不同硬件径号→时延/相位跳变）。纪律：跨快照**路径跟踪**（时延/角度/功率邻近匹配）→ 已有径保持槽位 → 新径入空槽（birth ramp）→ 死径先 ramp 归零再释放槽位、**槽位不立即复用**。此为《T1-05》§5-3（多快照序列）展开时的设计基线。（槽位释放/复用仅适用于本条参数流式面；A 档统一栅格不复用，Q2 裁定 2-③。）

## 3. Q2 · 按轨迹提取 TDL 如何处理？

**问**：RX/TX 沿轨迹移动、RT 产出多快照（或引擎产时变 CIR）时，如何提取并下发 TDL？

**已有约定**：
- 多快照序列是多普勒来源选项 (3)：「多个 MPDB 时刻 → `time.mode=time_varying`，由相位/位置演化估多普勒（A 档时间维前提）」，**本期 P1 只落选项 (1) 默认 0 与 (2) 几何速度注入** `f_d=(v·k̂)/λ`（《T1-05》§5）。
- M4 本期单快照 static；多快照/时变待 RT 侧形态确定后扩展（《T2-04》§8 开放问题 1）。
- 引擎侧已有时变通道：`want_cir` → `gains[n_pairs, n_snapshots, n_taps]`，落 `gain_series`/`cir_ref`（《T2-03》§2/§3）。

**★裁定（轨迹→TDL 的两档出口，设计基线）**：
1. **共用前置**：轨迹 = 快照序列（MPDB 多时刻或引擎 CIR）。逐快照跑既有单快照管线（清洗→导向→量化合并→选径），再做**跨快照路径跟踪 + 槽位稳定**（Q1 裁定 3）得到统一 tap 集合。
2. **A 档出口（时变，现实可用）**：统一**时延栅格**（跟踪后 tap 集合的并集，时延取跟踪代表值——快照间时延固定，与引擎侧「时变时延本期不支持」同口径《T2-03》§2）→ 逐时刻复增益填 `gain_series`（RT 的 H 自带传播相位，快照间相位差即多普勒，无需另行合成）→ `time_varying` canonical → AscCirBackend 出 `.asc`。三条配套裁定：
   - ① **并集超 24 的处置**：24 径上限是 **RF-SoC 设备约束**（`RFSOC_CAPS.max_paths=24`），不是 A 档管线约束——`.asc` 文件后端无径数上限（`ASC_CAPS.max_paths=None`，《T2-02》§2）。A 档统一栅格**不做 top-24 截断**（截断会静默丢多径能量），保留全部跟踪径；径数约束在 resolve 能力门按目标后端裁决（paths 超 max_paths → 拒，《T2-06》§5 既有设计）。与 schema `taps ≤24`（《T1-03c》§5.2/§9）的冲突登记为 **T1-03c 升版议题**：time_varying 载荷的 tap 上限应改由后端 capabilities 裁决（static 维持 ≤24）；放宽落地前，实现遇超 24 的 A 档模型**显式拒绝并报错**（不静默截断）。
   - ② **时延冻结的有效边界**：单径跨轨迹时延漂移 < 1 个硬件时延步长（(1000/120) ns）时冻结近似成立；超限由跟踪层**分裂为新径**（旧径 death、新径 birth，Q1 裁定 1）；冻结损失记 FidelityReport（与 B 档出口对称入账）。
   - ③ **统一栅格内槽位一经分配不复用**（一个 Tap = 一个时延 + 一条 `gain_series`）——Q1 裁定 3 的槽位释放/复用仅适用于参数流式面。
   采样充分性按 Q0 裁定 1 校验。
3. **B 档出口（准静态点列，当前硬件）**：轨迹退化为**快照点列回放**——逐快照产 static TDL artifact，按次序逐次 re-apply（每次全事务，衰落状态不承诺保持，Q0 裁定 2）；多普勒用几何速度注入（选项 2）让硬件内生衰落逼近轨迹动态。保真损失（点列非连续、切换瞬间视同重置）记 FidelityReport。
4. **实现节奏不变**：本期不落轨迹管线（P1 = 选项 1/2）；A 档出口随 A 档推进（《T1-13》）。本裁定只钉设计基线，防后续实现走「逐快照独立选径」的歧路。

## 4. Q3 · T2-M4 的簇合并是什么策略？

**问**：M4（MPDB 导入）的簇合并采用什么策略？

**答（澄清 + 既有约定汇编）**：**M4 没有「簇合并」**——聚簇（`reduce_RT_to_CDL`，射线→簇）归 M5，M4 本期仅透传 `CHANNEL_TYPE`（《T2-04》§8 开放问题 3）。M4 提供的是**射线→硬件时延 bin 的量化合并**纯函数（《T2-04》§5），策略五条：

1. **顺序纪律**：量化合并必须在 M5 施加导向相位**之后**调用——M4 只提供机制不自行调用，防「先合并后导向」破坏空间相关性。
2. **bin 划分**：`code = round(delay_s / PATH_DELAY_UNIT_S)`，步长 (1000/120) ns，码值 0..1050；超范围**丢弃并计数上报，不得夹到端点 bin**（夹取会把不存在的能量堆到 0/1050）。
3. **bin 内合并**：**恒为复增益相干叠加（保相位）**；`power_mode=noncoherent` 不改变叠加方式，仅用于功率统计。
4. **top-N 选径**：`select_strongest(bins, k≤24, power_mode)`，排序功率按 power_mode 取 coherent=|Σgain|²（相消干涉计入）或 noncoherent=Σ|gain|²——两者可选出不同径集，必须显式入参。
5. **归一化**：基准 `ref` 由 M5 传入（全系统共享，《T1-06》要点 2），不得自行取局部 max（旧 `tdl.py` 行为已废）。

**★裁定（精化文档空白——bin 内角度合并方法）**：《T1-05》§3 伪代码已示意 `angles = weighted_angles(grp)`（加权取向已定），《T2-04》§5 迁移清单只说「角度列全程保留」——**加权方法**两处均未定。裁定：**功率加权平均**——方位角（AoA/AoD az）按**圆统计**（加权复单位向量求辐角，避免 359°/1° 线性平均成 180°），天顶角（zen ∈ [0,180°]）线性加权即可。落 T2-04 下次修订。

> 注：《T1-05》§3 伪代码中 `power = ncoh if noncoherent else |coh|²` 的合并写法是早期表述；模块级精化后以《T2-04》§5 为准（合并恒相干、power_mode 只影响排序功率），见 §7 落点。

## 5. Q4 · 我们设计的 API 所处位置在哪里？

**问**：仪器行业的 API 能力通常「native inside」（SCPI 服务内嵌在仪器固件里），我们的 API 在哪？

**已有约定**：
- API 位于五层架构 **L4（API 网关层）**；所有前端只调用 L3 同一组服务接口（《T1-03》§2/§3）。
- 部署视图：CEP 主服务进程（FastAPI，监听 HTTP/SCPI-TCP）跑在**单机部署单元（外部控制机）**上，经 TCP 连 RF-SoC 设备；ChannelEgine 为同节点独立进程/容器微服务（《T1-03》§6）。
- 设备内只有**私有二进制 TLV 帧协议**（`FDB18540…AE86`），面向寄存器级控制，**不对外暴露**——只是 L1/L2 对接面（《T1-04》§1/§2）；对外 API 是平台自定义高层抽象，把寄存器语义提升为「信道/场景/会话」语义；canonical model 与设备协议是内部契约，不直接暴露给 API 版本（《T1-04》§7）。

**答（定位）**：**我们的 API 不在设备内，在设备之上的外置控制平面。** 与传统仪器「API 内嵌固件」的差异是刻意的：
1. **能力不在盒子里**：平台的核心能力（38.901 引擎、MPDB 导入、相关性合成、多后端渲染）依赖 numpy/PyTorch/微服务生态，天然在上位机；设备固件只承担实时衰落生成。
2. **多设备/多后端编排**必须站在单台设备之外（`ChannelAddress=(device_id,…)`，《T1-10》）。
3. **对第三方的「仪器体验」由 SCPI 兼容层在平台层复刻**（SCPI-over-TCP，M7 网关内，P4）——测试台看到的仍是「一台可 SCPI 的仪器」，只是 SCPI 服务点从固件搬到了控制平面。
4. **★澄清**：分层（L1–L5）与部署解耦——当前基线是外部控制机单机部署；若未来设备内嵌 Linux 有足够算力，把 CEP 主服务整体部署进设备属**部署形态选项**，不改变任何层间契约（无需文档变更，此处备案即可）。

## 6. Q5 · 我们的 SDK 跟 API 是什么关系？

**问**：我们的 SDK 跟 API 是个什么关系？

**已有约定**（《T2-09》《T2-07》《T1-04》）：
- **SDK 是 REST 的薄客户端封装**：`cep_sdk` 唯一上游是 M7 REST `/v1`（OpenAPI 契约），**不直连 L3**；独立包、不依赖服务端代码，传输仅 `httpx`；**零业务逻辑**（一切裁决在服务端），只做传输、映射与便捷语义（资源对象化、长任务等待、SSE 消费、错误类型化）。
- **合同单一真源**：OpenAPI 是合同；SDK 首版手写 + OpenAPI 黄金 diff 防漂移；版本对齐 `cep_sdk 1.x ↔ /v1`，首调惰性 `GET /v1/version` 校验。
- **能力一一对应**：SDK 语义与 REST 一一对应（《T1-04》§4），不多（无私有旁路）不少（REST 有则 SDK 有）；错误 problem+json → 类型化异常，映射表与 T2-07 §3 错误表同源维护；鉴权即 REST API-Key。
- **消费方边界**：SDK 服务第三方脚本/CI/未来 CLI；**GUI 直连 REST 不经 SDK**（M11 仅语义对照）；SCPI 是 M7 网关侧的另一前端，与 SDK 无关。

**答（一句话）**：**API 是合同，SDK 是该合同的 Python 便捷面。** 三个对外前端（REST 直连、SDK-over-REST、SCPI 转译）最终都收敛为同一组 L3 服务调用，杜绝逻辑分叉（《T1-04》§1 总则、《T2-07》§1）。
