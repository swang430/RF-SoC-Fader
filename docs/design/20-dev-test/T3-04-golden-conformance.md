## 1. 概述与定位

黄金基线是**设计的可执行形态**：设计文档说「应当如此」，黄金资产把「如此」冻结成字节/数值/文本，让 CI 逐次裁决。本篇按《T3-01》§4 通则把《T3-00》§5-A 的资产**逐项登记**：目录、出处、首录程序、对比判据、消费用例段；并定**重录（变更）流程**。

**非职责**：用例矩阵（→T3-02，本篇只给资产、不给用例判据）、存放通则本身（→T3-01）、阈值定标（→T3-05——黄金对比是「等/不等」，不含待定标阈值）。

**两条铁律**（《T3-00》原则 2 的实例化）：

1. **出处独立于实现**：每份资产的产生路径不得经过被测代码。生成脚本存 `tools/golden/`，只允许依赖 numpy/标准库，**禁止 import 平台包**（CI 静态检查强制）——防「实现生成黄金再验证实现」的自证回路。
2. **首录双确认**：手工转录类资产（docx hex/3GPP 表）首录 PR 需第二人对出处逐字节/逐行核对并在 PR 留痕；脚本推导类资产随 PR 附推导文档。

## 3. 变更（重录）纪律

| 步骤 | 要求 |
| :-- | :-- |
| 触发 | 仅限：出处方升版（docx 修订/引擎样例更新/3GPP 勘误）、冻结设计升版（如 T1-03c v1.2）、首录错误被证实 |
| PR | **独立 PR**（不与实现变更混提，《T3-00》§7 红线）；标题注明触发源；出处 diff 附于描述 |
| 版本 | 文件名 `v<n>` 递增；**旧版不删**（provenance.md 记录替换原因与日期）——历史 CI 结果可复核 |
| 评审 | 核对出处而非实现输出；手工转录类仍需双确认 |
| 联动 | 消费用例与追溯矩阵同 PR 更新（《T3-00》§4） |

## 4. 开放问题

1. **docx 示例覆盖缺口**：35 ID 中 docx 有完整 hex 示例的是少数——「按规格文本推导」的黄金帧在 HIL-C 首次真机会话时**逐条对表回验**（设备接受且回显一致），回验结果写回 provenance；在此之前这些条目标 `derived` 等级。
2. G3 引擎样例的版本锚定：ChannelEgine 独立仓演进后样例重生成的兼容策略（镜像格式版本主权在源契约，T2-10 §2）——随引擎服务层跨仓 PR（T2-03 §8-4）一并定。
3. G8 SCPI 黄金的指令子集随 P4 收敛（T2-07 §9-3）——先登记占位域。

## 5. 本篇验收

- G1–G11 每项五要素齐备（目录/出处/首录/判据/消费段），可直接开工首录；
- 两条铁律有 CI 强制点（`tools/golden/` import 检查、首录双确认留痕）；
- G7/G10 两类「非规格出处」资产（历史资产/幂等锚）的例外性质显式声明，不与规格类黄金混淆；
- 重录纪律五步可执行，且与追溯矩阵联动闭合。
