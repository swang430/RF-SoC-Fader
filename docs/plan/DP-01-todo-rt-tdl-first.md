## 1. 定位与裁定

**本文档是当前唯一的开发 TODO 台账**：条目在此登记、状态在此更新（经 PR）、完成在此销账（记 PR 号）。

**主线裁定（2026-07-19）**：执行序从「按阶段推进」细化为「按**产品切片**交付」——

1. **第一优先：RT-TDL 可用**（MPDB → TDL → 设备播放），且 **GUI 可控**；
2. **其次：输入输出外围配套**（仍围绕 RT-TDL 可用），使之成为**生产力工具**；
3. **同时：后续开发内容规范建账**，便于逐项展开。

与冻结基线的关系：

- 该主线与《T1-13》§4 关键路径（**旗舰优先**：RT→TDL 播放走 P0→P1 主线）一致；差异仅是把 **M7/M11 的最小子集提前**到第一切片（原属 P2「GUI 对接」）——这是**执行序调整**，不改变 P0–P4 的内容物与验收信号，验收仍按《T3-05》§4 逐阶段签发。
- 条目**不设计新东西**：每条实现依据均指向冻结条款；实现中发现设计错误，先走升版纪律改设计（先例 T1-03c v1.1→v1.2），再同 PR 改实现与测试（《T3-00》§4 设计即法）。
- **测试先行**（《T3-00》）：首个代码 PR 起启用 PR gate/CI gate（《T3-05》§2「P0 首日可启用」）；追溯矩阵载体 `obligations.yaml` 随首条目建立（《T3-01》§7）。

## 6. 与 gate/验收的衔接

- **PR gate / CI gate**：随 TD-A-01 首个代码 PR 启用（《T3-05》§2 试运行参数）；`obligations.yaml` 与四查脚本同 PR 就位（《T3-01》§7）。
- **里程碑签发**：MS-A ⇒ P0+P1 验收 gate（《T3-05》§4 前两行 + 矩阵快照归档）；MS-B 条目按各自 P 阶段归属签发，整体以 §4 使用者走查记录留档；MS-C 各项展开时在条目行补验收映射。
- **本文档维护**：条目状态变更随对应开发 PR 顺带更新（一行 diff）；纯状态更新不必单独开 PR。

## 7. 本篇验收

- 三个里程碑目标各可用一句话陈述，且第一切片（MS-A）每条目有冻结条款出处与可判定验收；
- 条目规范（§2）足以让后续新增条目不需再议格式；
- 全部 blocked 项均指向《T3-03》台账或硬件确认项，无凭空承诺（通用 fader 原则：当前硬件不支持 ≠ 平台不设计）。
